## 核心原则：一个月后测试

写完后问自己：如果一个月后、没有任何上下文地读这篇文档，能不能仅凭文档本身重建每个决策的 WHY？如果任何一步需要"你当时在场才能理解"，那就没写够。

## 必须避免的错误

| 错误 | 表现 | 问题 |
|------|------|------|
| 写成会议纪要 | "Round 1 讨论了 X，Round 2 讨论了 Y" | 读者看到流水账，不是决策逻辑 |
| 推理黑箱 | "分析后发现"、"推演到极致" | 结论有了，推理过程丢失 |
| 调研单独列章节 | 社区方案调研和设计推导分离 | 读者需要来回跳转理解"为什么选了这个" |
| 结论和过程重复 | 演进记录和推导链各写一遍 | 内容冗余 |

## 正确的结构

设计文档的"为什么"部分应该是一条因果链：

```
现状（带证据）
  → 问题是什么（可验证的事实，不是感觉）
    → 需要解决什么（逐项排查，排除不需要的）
      → [遇到解不了的问题] 社区怎么解决的（就地展开调研）
        → 为什么选了这个方案（对比表 + 排除理由）
          → 新方案引出了什么新问题
            → 新问题怎么解决的（继续展开推理）
              → 最终方案
```

## 具体手法

### 1. 现状要有证据
不要写"感觉不好用"，要写"审计发现 agent 从未调用过 memory_search — 实际使用量为零"。

### 2. 每个"分析后发现"都必须展开
- 排查选项 → 逐项排查表
- 理解时序 → 画时间线
- 评估方案 → 对比表 + 排除理由
- 决策选型 → 候选/采用/理由三列表

### 3. 调研嵌入到遇到问题的节点
不要单独列章节。在推导中遇到"需要看看别人怎么做"时就地展开：问题 → 调研对比表 → 选型决策表 → 继续推导。

### 4. 推导链形成闭环
最后用缩进的推导链总结全文因果，让读者 5 秒抓住完整逻辑。

## 一句话总结

设计文档不是记录"我们怎么讨论的"，而是记录"一个理性的人面对这些事实，为什么会一步步走到这个结论"。

**Why:** 在 memory v2 设计讨论中，前几版文档写成了会议纪要 + 推理黑箱，用户反复要求补充推理过程。最终版本通过逐项排查表、时间线分析、方案对比表展开每一步推理，用户确认满意。

**How to apply:** 所有涉及 brainstorming/spec 的设计文档，在写"为什么这么设计"时必须遵循此方法论。参考实例：`.dev-docs/versions/v0.0.17/memory-v2-analysis.md`。
